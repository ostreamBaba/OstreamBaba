# Adapter模式

*  在程序世界中，经常会存在现有的程序无法直接运行的情况，需要做出适当的变换后才可以使用的情况，这种用于填补“现有的程序”和“所需的程序”之间差异的设计模式就是Adapter模式(也称为wapper模式)。

* Adapter模式有一下两种
    * 类适配器模式(使用继承的适配器)
    * 对象适配器模式(使用委托的适配器)

```java
电源的比喻和示例程序的对应关系
           电源的比喻     实例程序
实际情况   交流100伏特    Banner类(showWithParen)
交换装置   适配器         PrintBanner类
需求      直流12伏特      Print接口(printWeak)
```


* 什么时候使用Adapter模式
    *  很多时候，我们并非从零编程，经常会使用到现有的类。当现有的类已经被充分测试过了，Bug很少了，而且已经被用于其他软件之中时，我们更愿意将这些类作为组件重复使用。Adapter模式会对现有的类进行适配，生产新的类。通过这种模式我们可以很方便地创建我们需要的方法群。(若出现bug时，只需要调查扮演adapter角色的类即可，现有类(adaptee角色)不需要调查)

    *  当现有的类适配新的接口(API)时，使用Adapter模式是理所当然的，让现有的类适配新的接口的(对已经测试完毕的现有代码的进行修改，就必须在修改后重新进行测试)。而Adapter模式可以在完全不需要修改现有代码的前提下使现有的类适配于新的接口。

    * 对于版本的升级和兼容：(提高于旧版本软件兼容性的Adapter模式)
        * 支持Version1.0的程序 ----- > MyClass Version1.0

        * 支持Version2.0的程序 ------> MyClass
        Version2.0
        * 支持Version1.0的程序 ------> 1.0 < ---- > 2.0（转换适配器） ----- >  MyClass Version2.0
        * 让新版本扮演adaptee的角色，旧版本扮演Target的角色，接着编写一个扮演Adapter角色的类，让它使用新版本的类来实现旧版本的类的方法。
    * 功能完全不同的类不能使用Adapter模式
